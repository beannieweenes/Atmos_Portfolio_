01 · Context

The situation I walked into

ADAFSA manages agricultural licensing, livestock management, food safety and fodder subsidies for Abu Dhabi's farming community — all through the UAE's TAMM government platform.

My position
Designer from Capgemini, embedded within the ADAFSA product team alongside two client-side designers
The setup
Three designers total. Approval and sign-off happened client-side. My job was to propose, justify, and build.
What existed
Working screens. Real users. No underlying system, no documentation, no shared language between design and dev.

The TAMM platform already had live services running. Users — many of them farmers with limited digital experience, accessing the platform in Arabic — were submitting monthly livestock reports, requesting subsidised fodder, and tracking application status through these screens.

The screens worked. But every new service was being built from scratch because nobody had documented what patterns already existed. The result: three different implementations of the same form input, status labels that meant different things on different pages, and Arabic RTL layouts that had been applied as an afterthought on some screens and ignored on others.

02 · The Problem

Three problems,
one root cause

The surface issues were all symptoms of the same thing: there was no shared design language. Every designer and developer was working from memory and personal interpretation.

01

Inconsistent UI across services

The same number stepper input had three different visual implementations. Status indicators used different colours and labels for identical states across different services.

02

No documentation or handoff process

There was no component documentation, no token naming, no usage guidance. Developers were eyeballing designs and making decisions that should have been specified. Every handoff was a conversation.

03

RTL Arabic support was broken

Arabic layouts had been added reactively rather than designed for. Form inputs, the process stepper, and navigation elements didn't mirror correctly — a critical issue for a government service used primarily by Arabic speakers.

What made this complex was the team structure. With two client-side designers who had existing opinions about how things should look, and developers who needed clarity to build, I had to make the case for system thinking without having authority to mandate it. Every decision needed to be collaborative and well-reasoned.

03 · My Process

How I approached
building the system

I didn't start by designing new things. I started by understanding what already existed — because the goal was to document and systematise reality, not impose a new aesthetic.

1
Audit

Inventory every screen and pattern

I went through all existing ADAFSA services on TAMM and catalogued every UI element — every button, every input, every status indicator. The goal was to see what patterns actually existed before deciding what to standardise. This produced a list of 20 component types with 87 combined variants.

2
Extract

Pull out the underlying tokens

I mapped every hardcoded colour value, spacing unit and font size to a named token. The colour palette went from 34 unique hex values scattered across files down to 9 semantic tokens — primary, neutral, and semantic (success, error, warning, info). This was the foundation everything else would reference.

3
Align

Get the client team on the same page

Before building anything, I presented the audit findings to the two client designers. Showing them the inconsistencies side by side — three steppers, four status label variants — made the case for standardisation more effectively than any argument could. We aligned on the canonical version of each component together.

4
Build

Design the component library

With alignment in place, I built the component library in Figma — structured in layers: tokens, base components, composite components, page templates. Every component was built with all states (default, hover, focus, error, disabled) and RTL variants designed alongside the LTR version, not after.

5
Document

Write usage guidance for every component

A component library without documentation is just a collection of shapes. For each component I wrote: what it's for, when to use it, when not to, which token it references, and how it behaves in RTL. This was the piece the team had never had before — and the most time-consuming part of the work.

04 · Key Decisions

The choices that
defined the system

Design systems are made of decisions. Here are the ones that mattered most on this project, and the reasoning behind each.

🟡
Color

Amber as the primary CTA, not navy

The ADAFSA brand is navy-led, but I chose amber (#F5A623) as the primary action colour. Navy on white creates sufficient contrast but reads as informational — amber creates urgency and focus on the action. More importantly, amber maintains contrast against both white backgrounds and navy sidebar panels, making it work across all layout contexts. This was a small change with significant impact on task completion clarity.

📐
Interaction Design

Oversized touch targets on the number stepper

The number stepper is the most-used input on the fodder subsidy form — farmers are entering counts for dozens of animals across multiple categories. I designed the minus and plus buttons at 40×40px, significantly larger than standard. The primary users are farmers, often on mobile, sometimes in outdoor conditions. Small touch targets on a high-frequency input would create real friction for real people.

🔢
Navigation Pattern

The process stepper as the platform's primary nav metaphor

Every ADAFSA service follows the same multi-step pattern. I standardised the vertical process stepper — with numbered steps, three states (pending, active, complete), and sub-step support — as the single navigation pattern for all services. This meant users only need to learn how it works once. It also gave developers a single, well-specified component to build, rather than reimplementing step navigation for every new service.

🌐
Internationalisation

RTL as a first-class concern, not an afterthought

The previous approach to Arabic was to flip the layout at the end. This consistently broke — particularly the process stepper, form inputs, and the breadcrumb navigation. I rebuilt every component using logical properties (start/end rather than left/right) so RTL is structural, not cosmetic. The stepper numbers re-sequence correctly, form labels align properly, and the breadcrumb reads right-to-left without manual overrides.

📝
Documentation

Four status states, clearly named, strictly used

Before the system, ADAFSA services used "In Progress", "Pending", "Submitted", "Under Review" and "Processing" — sometimes interchangeably, sometimes to mean different things. I reduced this to four named states: Current Step, Pending, Completed, Declined. Each has a defined meaning, a distinct colour token, and a usage rule. Clarity in naming is clarity in the user's mental model.

06 · Reflection

What I learned

Three things this project taught me that I carry into every project now.

🏗️   Systems work is political, not just technical

Getting two client-side designers to adopt a shared system required showing, not telling. The audit — seeing 34 colour values listed side by side — made the case in a way that arguments couldn't. The lesson: make problems visible before proposing solutions.

🌐   RTL is a design decision, not a dev task

Arabic layout support fails when designers treat it as a developer concern. Every component needs to be conceived bidirectionally from the start. I now design with logical properties in mind from the first frame, not after everything is built.

📝   Naming is design

Standardising the four status states had as much user impact as any visual change. When the system uses clear, consistent language, the product does too. Good naming reduces user confusion without changing a single pixel.

📊   Not measuring outcomes is a gap to close

This project was delivered without impact metrics. The work improved consistency and velocity — I know this from team feedback — but I couldn't quantify it. On future projects, I build in success criteria at the start so the outcome story can be told with evidence, not just assertion.
